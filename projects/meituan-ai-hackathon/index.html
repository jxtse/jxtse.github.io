步语 BuYu 面向「周末半天不知道去哪儿」这一高频本地生活场景，把一句含糊需求转化为一条可确认、可解释、可执行的本地生活方案。它不只是生成一段推荐文案，而是让 LLM 在真实候选、实时可用性和用户反馈之间做选择：先理解需求，再调用工具查证，最后把推荐、确认与执行连成闭环。本文聚焦比赛要求的三件事：Planning 策略、工具调用链路、异常处理机制；并说明通用规划能力如何通过 LoRA 内化。

01Planning 策略

规划不是一次性吐出整张行程，而是 LLM 作为 tools orchestrator，在 ReAct 循环（Reason → Act → Observe）里分段、软固定、渐进式确认地推进。每一段都是一轮「Reason（结合当前会话上下文想清楚这段要什么）→ Act（按需调用地图定位、地点检索、详情、团购、评论、种草帖、查位等工具）→ Observe（读取结果，必要时调整选择）」。接受、换选、追问、查位和下单都被视作同一条规划链上的状态变化；每一次用户反馈都会更新当前任务状态摘要，让下一段推荐继续沿着本次出行的真实意图收敛：

stage · locate定位

读取当前位置与商圈，作为后续距离计算的原点。

stage · intent意图解析

自然语言 → 结构化约束：场景 / 人群 / 饮食 / 时间窗 / 预算，更新当前任务状态。

LLM · JSONAgent 推理

软固定三段框架

  • play → eat → extra：默认「先玩、再吃、轻松收尾」。段的顺序即推荐顺序。
  • 场景自适应：约会把 extra 设计成「鲜花惊喜」，家庭设为「亲子手作 / 环湖消食」，朋友设为「续摊收尾」。
  • 可裁剪：时间紧（hours≤3）砍掉 extra，只想吃饭则只留 eat。
  • 动线感知：每段决策都把「从上一站过去的摩擦成本」纳入，下一张卡不是孤立推荐，而是接住整条路线的后续选择。

渐进式总结 + 会话上下文

  • 每段出一张卡，用户只需接受 / 换一个 / 看详情三选一，不必从长列表里挑。
  • 每次选择都更新上下文：accept → 当前卡写入行程并沉淀为偏好信号，推进下一段；reject → 将该 id 加入本轮排除集，同段换次优候选，下段也据此调整。
  • 多轮对话渐进式总结：用户补充偏好或质疑推荐，Agent 把新信息并入当前任务状态，判定 keep / switch / refine 并更新约束，让后续推荐更贴近本次需求。
  • 推理可见：每一步通过 SSE 推 thinking / stage / tool / self_heal 事件，右侧链路实时展示 Agent「为什么这么选」。其中 自动调整（internal: self_heal）事件解释每一次实时改选。

02工具调用链路

外部能力被拆成一组可审计的工具（name / description / parameters / invoke()）。LLM 不直接编造商户或订单，而是在 ReAct 循环里读工具描述、调用工具、查看返回值，再决定下一步。链路覆盖定位、地点搜索、地点详情、团购套餐、用户评论、种草帖、实时排队、预订取号和下单执行。当前实现使用完整 Mock API：dianping_search 负责从南京代表性地点库召回候选，dianping_detail 返回图片、套餐、评论、亮点和同段备选，meituan_query_queue 返回排队 / 可订状态，执行类工具完成订座、买券、取号或下单并返回确认号。右侧 Agent 链路由事件总线驱动；公网环境下，为避免 SSE 被代理缓冲，关键预设和用户点击动作会随接口响应返回 agent_events，由前端按时间线回放。商户数据全部走 Mock 美团 / 点评 层，不产生真实交易，便于评审复现。

阶段工具角色输入 → 输出
定位meituan_locate地图工具∅ → 当前坐标 + 商圈，作为距离原点
理解intent_parse (LLM)Agent 推理query → 结构化约束 JSON，更新当前任务状态（失败则启发式兜底）

Agent 与工具的边界：工具负责确定性操作——召回真实存在的候选、计算距离、匹配团购、查询实时队列、执行下单并返回可审计结果。 LLM 负责工具编排与取舍：读工具返回的 observation，结合当前会话里的偏好和排除集，在真实候选里（被严格约束「只能选一个，绝不编造」）综合 同行人结构、时间与体力状态、实时约束、体验风险和执行确定性，挑出最合适的候选并给出用户能理解的理由，再决定下一步调哪个工具。

03实时调整机制

规划与执行中遇到的落地障碍，目标是不中断流程、当场自动调整，并把每一次调整显式推给用户（自动调整事件，internal: self_heal），可见、可解释。机制贯穿规划期（风险前置）与执行期（逐站调整）两个阶段。

类型触发条件处理策略发生阶段
模型降级意图解析 / 对话的 LLM 调用异常或返回非法 JSON捕获异常 → 切换启发式规则兜底（关键词判场景/人数），流程继续不报错规划期
无座 no_table下单前查位无可订桌 / 等待 ≥30min，或下单返回满座出卡前预过滤降权高风险店；执行期则在同段候选自动改约，保留时间轴规划 + 执行
无票 sold_out下单返回票 / 名额售罄同段顺路候选自动替换重试，沿用已选时间，成功即写回行程执行期
时间冲突 conflict同一时段已有预约自动错峰 +15min 重排该站，避免撞车执行期
无候选某段在场景约束下检索为空放宽场景再搜；仍为空则跳过该段，不阻塞整体方案规划期
典型链路 · 家庭晚饭段实时调整
用户确认 Kiddo Lab（亲子手作）后，Agent 进入吃饭段。系统先按距离和评价召回 Green Bowl——它离上一站最近，也符合轻食需求； 但在下单前调用 meituan_query_queue，发现当前无可订桌位、预计等待 35min。 Agent 读到这条 observation 后重新推理：5 岁孩子刚完成一场高参与度活动、体力已下降，继续等位会显著放大「孩子哭闹、行程失控」的体验风险， 因此主动降权 Green Bowl，改选 Sprout Table。虽然 Sprout Table 距离更远，但有可订座位、等待更短、热食与轻食选择更稳， 更贴合「省心吃顿好的」目标。整个调整推 自动调整 事件呈现在右侧链路，用户全程可见为什么改。

规划期：风险前置

出卡前即调用查位工具，对「评分高但实时落地风险高」的餐厅主动降权， 优先把可订、等待短的候选放进卡片——避免把一个看起来不错、到店却坐不下的选项推给用户。 每一次降权都附带一段思考，解释「我原本会选 A，但实时队列显示它没桌，所以改 B」。

执行期：逐站调整

一键执行时逐站下单，命中无座 / 无票立即在同段合法候选里改约重试， 命中时间冲突自动错峰；每次调整推 自动调整（internal: self_heal → self_heal_ok）两段事件， 告诉用户「原选满座，已自动改为 X」。整条链路 Pass@1，不回退到「请稍后再试」。

04上下文与技能内化

步语 BuYu 不在工具层单独维护长期状态。单个会话里的偏好、排除集和已确认行程直接保存在当前任务上下文中，保证多轮对话能接住用户刚刚补充的信息；跨会话的通用规划能力则通过 LoRA / 参数化技能内化沉淀到模型能力里。分工原则很简单：稳定、可复用的规划模式适合内化；实时、精确、易变的事实留给工具检索。

通用技能内化 → 认知基础

  • 把通用技能内化进参数：分段规划范式、动线与体力常识、五类自动调整策略这类跨场景复用的「策略模式」，通过 LoRA + RL 内化为模型能力，而非每次写入上下文。
  • 先有基础策略，再做细粒度调用：Agent 需要知道怎样拆段、怎样处理动线和风险，才谈得上灵活调用地点、排队、下单等工具。
  • 避免把所有规则塞进 prompt：通用策略内化后，运行时上下文只保留本次任务真正需要的信息，减少提示膨胀，也让多轮对话更稳定。

结构化事实 → 工具实时检索

  • 商户事实不靠模型背诵：地点、套餐、库存、排队、价格都由工具侧 Mock API 返回，方便更新，也方便评审复现。
  • LoRA 更适合内化规律：它沉淀的是「怎么规划、怎么权衡、怎么自动调整」这类可复用模式，而不是逐字记住某家店今天有没有桌。
  • 落到步语 BuYu 的分工：稳定策略进入模型能力，实时事实保留在工具侧检索。这样既减少上下文负担，也避免把易变数据写死。
设计依据：步语 BuYu 把上下文和长期能力拆开处理：本次对话里的约束直接放在会话状态里，跨场景复用的规划能力通过 LoRA 内化，商户、排队、库存、价格等事实始终由工具返回。这样评审看到的不是一个靠提示词硬撑的推荐器，而是一个能在「理解需求 → 调工具 → 读返回值 → 改选择 → 执行」之间闭环运转的 Agent。
